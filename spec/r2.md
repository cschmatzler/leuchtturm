# R2 mail content spec

## Summary

Move canonical email content out of Postgres and into R2.

Keep Postgres + Zero for:

- accounts, folders, labels
- conversations
- messages
- participants
- unread/starred/draft flags
- snippets/previews
- search documents
- object pointers, checksums, and version metadata

Move to R2:

- raw provider payloads / raw MIME
- full rendered message bodies
- full rendered conversation bodies

This is a direct replacement, not a staged rollout.

- no dual-read path
- no dual-write path
- no cutover phases
- no fallback to Postgres body reads

When this work lands, mail body content is read from R2 and not from Zero.

## Decisions

### Attachments

Ignore attachments in this spec.

- attachment metadata can stay in Postgres
- attachment bytes are out of scope
- inline `cid:` attachment rendering is out of scope

### Sanitization

Sanitize on the server, not on the client.

Reasoning:

- fully expanded threads mean many message bodies are rendered at once
- preloading before render means client-side sanitization would sit directly on the thread-open hot path
- server-side sanitization lets us pay the cost once at ingest or rebuild time
- the client gets deterministic, already-safe HTML and only has to render it
- raw source still lives in R2, so parser/sanitizer changes can rebuild render bundles later without a provider re-sync

This does mean sanitizer changes require regenerating stored render bundles. That tradeoff is accepted.

### Thread loading

Threads stay:

- fully expanded
- fully preloaded
- unbounded

That means thread-open latency will still scale with total thread bytes. The goal here is to make that path as cheap as possible:

- no body content in Zero
- no client-side sanitization
- one conversation render fetch instead of one fetch per message

## Goals

- remove canonical email content from Postgres
- keep Zero focused on relational metadata
- make thread open fast for multi-tenant use
- preserve search in Postgres
- preserve the ability to rebuild render output from raw source

## Non-goals

- moving attachment bytes to R2
- partial or lazy thread loading
- capped thread sizes
- backwards compatibility with the old Postgres body read path
- storing multiple hot-path render formats just in case

## Storage model

## Postgres + Zero

Postgres remains the source of truth for metadata and derived projections.

Keep:

- `mail_account`
- `mail_folder`
- `mail_label`
- `mail_conversation`
- `mail_message`
- `mail_message_participant`
- `mail_attachment` metadata only
- `mail_search_document`
- `mail_message_source`
- new conversation render pointer metadata

Important nuance:

- `subject`, `snippet`, flags, participants, and search text remain in Postgres
- canonical raw content and canonical renderable content move to R2

## R2

R2 stores canonical email content.

Per message:

- raw source object
- message render bundle

Per conversation:

- conversation render bundle

## Render artifacts

### Message render bundle

Used for:

- standalone message page
- flat-mailbox providers
- rebuild/debugging

Shape:

```ts
type MessageRenderBundle = {
	version: 1;
	messageId: string;
	preferredKind: "html" | "text";
	html: string | null;
	text: string | null;
	parserVersion: string;
	sanitizerVersion: string;
};
```

Notes:

- `html` is already sanitized
- `text` is plain text fallback
- both can exist, but the preferred kind decides normal render behavior

### Conversation render bundle

This is the hot read artifact for threaded mail.

Used for:

- fully expanded thread preload
- one-shot conversation body fetch

Shape:

```ts
type ConversationRenderBundle = {
	version: 1;
	conversationId: string;
	parserVersion: string;
	sanitizerVersion: string;
	messages: Array<{
		messageId: string;
		renderKind: "html" | "text";
		content: string;
	}>;
};
```

Notes:

- store only the preferred render content per message in the conversation bundle
- do not store both html and text for every message in the conversation bundle
- keep message order aligned with thread order

## Schema changes

### `mail_message_source`

Use `mail_message_source` as the canonical pointer table for message-scoped content in R2.

Supported `sourceKind` values should include:

- `raw_mime`
- `gmail_full_message`
- `render_bundle`

Each row stores:

- `storageKind = 'r2'`
- `storageKey`
- `contentSha256`
- `byteSize`
- `parserVersion`
- `sanitizerVersion`
- `encryptionMetadata`

### New conversation render pointer table

Add a new table for conversation-scoped render bundles.

Suggested name:

- `mail_conversation_render`

Fields:

- `conversationId`
- `accountId`
- `userId`
- `storageKind`
- `storageKey`
- `contentSha256`
- `byteSize`
- `messageCount`
- `parserVersion`
- `sanitizerVersion`
- `createdAt`
- `updatedAt`

This table exists so the thread-open hot path is explicitly conversation-scoped instead of being hacked through message-level source rows.

### `mail_message_body_part`

Remove `mail_message_body_part` from the product architecture.

- stop writing body content there
- stop reading body content there
- remove it once the new implementation is in place

### `mail_provider_state`

Keep `mail_provider_state` only for sync state.

Allowed contents:

- cursors
- checkpoints
- folder/account sync bookkeeping

Do not store per-message provider payloads there anymore.

## Write path

For each ingested message:

1. fetch raw provider payload or raw MIME
2. parse the message
3. derive message metadata
   - subject
   - snippet
   - flags
   - participants
   - `hasHtml` / `hasPlainText`
4. derive search text
5. sanitize HTML on the server
6. build the message render bundle
7. upload the raw source object to R2
8. upload the message render bundle to R2
9. rebuild the conversation render bundle for the message's conversation
10. upload the conversation render bundle to R2
11. write Postgres metadata and pointer rows
12. update `mail_search_document`

Accepted tradeoff:

- when a conversation changes, rewrite the whole conversation render bundle

That is intentionally simple and matches the read-optimized goal of this spec.

## Read path

## Metadata

Keep the existing Zero preload model for metadata.

Thread route preload continues to use:

- `zero.preload(queries.mailConversation({ conversationId }))`
- `zero.preload(queries.mailConversationMessages({ conversationId }))`

## Conversation content

Add one conversation render query outside Zero.

Endpoint:

- `GET /api/mail/conversations/:conversationId/render`

Behavior:

1. authenticate the user
2. authorize access to the conversation through Postgres metadata
3. load `mail_conversation_render`
4. fetch the render bundle from R2
5. return the bundle

Route loader behavior:

- preload Zero metadata
- preload the conversation render bundle with the non-Zero query layer

The conversation page should render from:

- Zero metadata
- one preloaded conversation render payload

No lazy body loading.

## Message content

Add a single-message render endpoint for standalone message pages.

Endpoint:

- `GET /api/mail/messages/:messageId/render`

Behavior:

1. authenticate the user
2. authorize access to the message through Postgres metadata
3. resolve the message render bundle pointer from `mail_message_source`
4. fetch the render bundle from R2
5. return the bundle

## Client changes

## Threaded conversation page

The conversation page should:

- preload conversation metadata through Zero
- preload the conversation render bundle through the regular query client
- render all message bodies from the conversation render bundle

Each message row should look up its render content by `messageId` from the already-loaded conversation bundle.

## Message page

The message page should load the standalone message render bundle instead of querying body parts through Zero.

## Remove client-side sanitization from the hot path

Mail rendering should no longer sanitize HTML in the browser during normal display.

- `sanitizeEmailHtml(...)` should not be part of normal mail body rendering
- the client should render already-sanitized HTML returned from the server

## Search

Keep search in Postgres.

`mail_search_document` remains the searchable projection and is rebuilt from parsed content during ingest.

Search should never hit R2 on the hot path.

Accepted duplication:

- renderable content in R2
- searchable projection in Postgres

That duplication is intentional.

## Compression

Because threads are unbounded, conversation render bundles must be served compressed.

Minimum requirement:

- JSON response
- HTTP compression enabled

This is enough for the first implementation.

## Security

- R2 bucket is private
- render bundles are not public objects
- app endpoints perform authentication and authorization
- object keys are internal and opaque

## Implementation tasks

This is one body of work. Do all of it together.

1. add the R2 storage abstraction used by mail ingest and mail reads
2. add the conversation render pointer table
3. extend `mail_message_source` usage for raw source and message render bundle pointers
4. update ingest to write raw source objects to R2
5. update ingest to sanitize HTML on the server
6. update ingest to write message render bundles to R2
7. update ingest to rebuild and write conversation render bundles to R2
8. update ingest to stop writing message body content to Postgres
9. update provider-state writes so per-message payloads are no longer stored in `mail_provider_state`
10. add the conversation render endpoint
11. add the message render endpoint
12. update the conversation route loader to preload the conversation render bundle
13. update the message route loader to preload the message render bundle
14. replace `mailMessageBodyParts` usage in the UI with render bundle reads
15. remove client-side sanitization from normal mail rendering
16. move existing stored message content into R2-backed raw/message/conversation artifacts
17. remove obsolete Postgres body-content storage once the new implementation is in place

## Success criteria

- Zero only carries metadata, not message bodies
- opening a thread preloads:
  - conversation metadata through Zero
  - one conversation render payload outside Zero
- the thread page renders fully expanded without lazy body fetches
- the browser does not sanitize mail HTML on the hot path
- Postgres no longer stores canonical email body content
- raw source remains available for rebuilds

## Accepted tradeoffs

These are intentional consequences of this design:

- thread-open latency still scales with total thread bytes
- changing one message in a conversation rewrites the full conversation render bundle
- giant threads can still be slow if they are huge enough
- attachments and inline attachment rendering are not handled by this spec
